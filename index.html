Overview

The Rackspace Private Cloud powered by OpenStack™ uses the openstack-ansible project to configure and install an OpenStack based private cloud. The Rackspace Private Cloud offering can be run in your datacenter, or ours, so we are provided this tool to generate a configuration that is based on your environment and will enable you to deploy your own OpenStack™ cloud configured to our OpenStack™ experts standards. We collect a little bit of information from you at the beginning of this wizard so that we can send you the output of this configuration generator as well as instructions on how to install your OpenStack cloud.

The information we collect will also be used to follow up with you to ensure you had a great experience installing the software and explain how we can show you that we are the world's foremost experts in operating OpenStack™ clouds.



Prerequisites

Rackspace Private Cloud Standard Reference Architecture



Based on this Reference Architecture you will need to have the following available to you before using this tool

  • 3 Servers, with Ubuntu 14.04 installed and an ip address. These servers will be responsible for running all of the OpenStack™ and ancillary services
  • 1 Server, with Ubuntu 14.04 installed and an ip address. This server will have elasticsearch, logstash, and kibana installed to centralize all of the logs from the environment.
  • 2 Servers, with Ubuntu 14.04 installed and an ip address, for Compute services
    • Note This configurator will also provide you the option to generate an "all in one" deployment if you are only interested in exploring OpenStack™
  • Either a softoware or hardware load balancer. If you have a software load balancer it will need to run on a server separate from where the OpenStack™ services will be run.
  • 3 RFC1918 Networks for the containers network, the overlay network, and the storage network.
    • If you have already used ip's in these ranges please have those available as well

Step 2: Network Information

The Rackspace Private Cloud deployment of OpenStack™ requires access to three networks. Typically these ranges are:
  • The Management Network (Control Plane for Infrastructure Services, OpenStack™ API's, and the OpenStack™ Dashboard
  • The Tunnel Network (Data Plane for project (tenant) VXLAN overlay networks
  • The Storage Network (Data Plane for storage services such as OpenStack™ block and object services.
Please enter CIDR ranges in each of the following input areas. If you do not enter anything the defaults in each box will be used.



Because the OpenStack™ API's are spread across three servers in the environment we require a load balancer to be part of the network. Its purpose is to balance requests to all of the API services and the OpenStack™ dashboard and ensure that in the failure of a service traffic can be balanced elsewhere.



Step 3: Infrastructure Hosts Information

In order to provide a highly available API infrastructure the Rackspace Private Cloud requires that there are three copies of all the OpenStack™ services running. This is to allow a failure in one or two of the infrastructure hosts while still maintaining availability of the OpenStack™ services. In addition to the OpenStack™ services these hosts will also house a three node galera database cluster to allow redundancy of information and a three node rabbitmq messaging and notification service required to pass information amongst the OpenStack™ services. Here is a complete list of the services that will be installed inside of lxc containers on these hosts:

  • OpenStack™ Identity
  • OpenStack™ Compute
  • OpenStack™ Networking
  • OpenStack™ Block Storage
  • OpenStack™ Image Service
  • OpenStack™ Orchestration
  • OpenStack™ Dashboard
  • MariaDB Galera
  • RabbitMQ

    • Additional OpenStack™ services, such as Object Storage, can be installed using the software, however, are out of scope for this tool.
